Core Concepts

Yeh terms baar baar aayenge — ek baar clear kar lo, poora roadmap easy lagega.

TermMatlab
Authentication"Tum ho kaun?" — identity verify karna (login).
Authorization"Tumhe yeh karne ki permission hai kya?" — roles/permissions check.
StatefulServer ko yaad rakhna padta hai kaun logged in hai (sessions, DB lookups).
StatelessServer ko kuch yaad nahi rakhna — token khud mein saari info carry karta hai (JWT).
HashingOne-way — password ko hash karte ho (bcrypt/argon2), wapas nikaal nahi sakte. Passwords ke liye.
EncryptionTwo-way — encrypt karke decrypt bhi kar sakte ho. Secrets/data ke liye, passwords ke liye nahi.
Golden rule

Password kabhi plaintext store mat karo. Hamesha bcrypt ya argon2 se hash karo, aur kabhi apna khud ka hashing algorithm mat banao.

01
Phase 1 · Learning only

HTTP Basic Auth

Kab use karo: sirf seekhne ke liye, ya internal admin tools/scripts jinhe koi UI nahi chahiye. Public-facing app ke liye kabhi nahi.

Sabse simple form: browser username:password ko base64 encode karke har request ke Authorization header mein bhejta hai. Server decode karke check karta hai.

middlewares/basicAuth.ts
import { Request, Response, NextFunction } from "express";

export function basicAuth(req: Request, res: Response, next: NextFunction) {
  const header = req.headers.authorization; // "Basic base64(user:pass)"
  if (!header?.startsWith("Basic ")) {
    return res.status(401).set("WWW-Authenticate", "Basic").end();
  }

  const decoded = Buffer.from(header.split(" ")[1], "base64").toString();
  const [username, password] = decoded.split(":");

  if (username === process.env.ADMIN_USER && password === process.env.ADMIN_PASS) {
    return next();
  }
  return res.status(401).json({ message: "Invalid credentials" });
}
Why we move on

Credentials har request mein dobara bhejni padti hain, koi expiry nahi, koi logout concept nahi, aur agar HTTPS na ho toh credentials plain dikh jaate hain. Isliye sirf learning ya trusted internal tools tak simit rakho.

02
Phase 2 · Stateful

Session-Based Auth (Cookies)

Kab use karo: classic server-rendered apps (EJS/Pug templates), ya jab tumhe har user ko instantly server-side logout/kick karna ho.

Login hone par server ek session create karta hai (DB/memory mein store), aur user ko sirf ek random sessionId cookie mein deta hai. Har request pe cookie aati hai, server session lookup karke user identify karta hai.

server.ts
import session from "express-session";
import connectPgSimple from "connect-pg-simple";

app.use(session({
  store: new (connectPgSimple(session))({ conString: process.env.DATABASE_URL }),
  secret: process.env.SESSION_SECRET!,
  resave: false,
  saveUninitialized: false,
  cookie: {
    httpOnly: true,      // JS can't read it -> stops XSS token theft
    secure: true,        // HTTPS only
    sameSite: "lax",
    maxAge: 1000 * 60 * 60 * 24 * 7, // 7 days
  },
}));

// after verifying password:
req.session.userId = user.id;

Instant logout / revoke✔ trivial — delete row from session store
Scales across microservices✘ every service needs the shared session store
Good fit forMonoliths, server-rendered pages, admin panels
03
Phase 3 · Stateless (naive)

Single JWT Auth

Yeh phase samajhna zaroori hai kyunki iski hi problems solve karne ke liye Phase 4 (access + refresh) bana hai.

Login pe server ek signed JWT deta hai jisme user id + roles hote hain. Server ko kuch store nahi karna — signature verify karke trust kar leta hai. Simple, lekin do badi problems:

  • Long expiry (jaise 7 din) rakho toh — token chori hone par attacker 7 din tak full access rakhta hai, aur tum revoke nahi kar sakte kyunki server kuch store hi nahi karta.
  • Short expiry rakho toh — user baar baar login karta rahega, bura UX.
utils/jwt.ts
import jwt from "jsonwebtoken";

export function signSingleToken(userId: string) {
  return jwt.sign({ sub: userId }, process.env.JWT_SECRET!, { expiresIn: "7d" });
}
// Problem: agar yeh token leak ho jaye, revoke karne ka koi tareeka nahi hai.
The insight

Solution ek hi long-lived token nahi, do tokens hain — ek short-lived jo kaam karta hai, ek long-lived jo sirf naya short-lived token dilwane ke liye use hota hai aur revoke bhi ho sakta hai. Yehi Phase 4 hai.

04
Phase 4 · Today's target — production ready for monolith / small scale

2-Token System: Access + Refresh

Yeh wahi setup hai jo tum deploy karne wale ho. Poora 5-layer architecture ke saath, taaki baad mein rewrite na karna pade.

Access Token — 15 min valid, JWT, user id + roles carry karta hai, memory mein (JS variable) rehta hai frontend pe, kabhi localStorage mein nahi.
Refresh Token — 7 din valid, random UUID, DB mein stored, HttpOnly + Secure + SameSite=Strict cookie mein bheja jaata hai. JS isko chhoo bhi nahi sakta — XSS se safe.

Flow

Login → access token (body) + refresh cookie set → access expire hone par 401 → frontend auto /refresh call karta hai → server cookie se UUID nikaal ke DB check karta hai → naya access token deta hai. Logout = DB se refresh token delete + cookie clear.

Prisma Model — Layer 5

prisma/schema.prisma
model User {
  id           String    @id @default(uuid())
  email        String    @unique
  passwordHash String
  role         String    @default("user")
  sessions     Session[]
  createdAt    DateTime  @default(now())
}

model Session {
  id           String   @id @default(uuid())   // this IS the refresh token value
  userId       String
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  expiresAt    DateTime
  createdAt    DateTime @default(now())
}

Repository — Layer 4 (only DB queries, no logic)

modules/auth/auth.repository.ts
import { PrismaClient } from "@prisma/client";

export class AuthRepository {
  constructor(private prisma: PrismaClient) {}

  findUserByEmail(email: string) {
    return this.prisma.user.findUnique({ where: { email } });
  }

  createUser(email: string, passwordHash: string) {
    return this.prisma.user.create({ data: { email, passwordHash } });
  }

  createSession(userId: string, expiresAt: Date) {
    return this.prisma.session.create({ data: { userId, expiresAt } });
  }

  findSessionById(id: string) {
    return this.prisma.session.findUnique({ where: { id } });
  }

  deleteSession(id: string) {
    return this.prisma.session.delete({ where: { id } });
  }
}

Service — Layer 3 (all business logic lives here)

modules/auth/auth.service.ts
import bcrypt from "bcrypt";
import jwt from "jsonwebtoken";
import { AuthRepository } from "./auth.repository";

const REFRESH_TTL_DAYS = 7;

export class AuthService {
  constructor(private repo: AuthRepository) {}

  async register(email: string, password: string) {
    const passwordHash = await bcrypt.hash(password, 12);
    return this.repo.createUser(email, passwordHash);
  }

  async login(email: string, password: string) {
    const user = await this.repo.findUserByEmail(email);
    if (!user) throw new Error("INVALID_CREDENTIALS");

    const valid = await bcrypt.compare(password, user.passwordHash);
    if (!valid) throw new Error("INVALID_CREDENTIALS");

    const expiresAt = new Date(Date.now() + REFRESH_TTL_DAYS * 24 * 60 * 60 * 1000);
    const session = await this.repo.createSession(user.id, expiresAt);

    const accessToken = jwt.sign(
      { sub: user.id, role: user.role },
      process.env.JWT_ACCESS_SECRET!,
      { expiresIn: "15m" }
    );

    return { accessToken, refreshToken: session.id, user };
  }

  async refresh(refreshToken: string) {
    const session = await this.repo.findSessionById(refreshToken);
    if (!session || session.expiresAt < new Date()) {
      throw new Error("INVALID_SESSION");
    }
    const accessToken = jwt.sign(
      { sub: session.userId },
      process.env.JWT_ACCESS_SECRET!,
      { expiresIn: "15m" }
    );
    return accessToken;
  }

  logout(refreshToken: string) {
    return this.repo.deleteSession(refreshToken);
  }
}

Controller — Layer 2 (req/res only, zero business logic)

modules/auth/auth.controller.ts
import { Request, Response } from "express";
import { AuthService } from "./auth.service";

const COOKIE_OPTS = {
  httpOnly: true,
  secure: true,
  sameSite: "strict" as const,
  maxAge: 7 * 24 * 60 * 60 * 1000,
  path: "/api/auth/refresh",
};

export class AuthController {
  constructor(private service: AuthService) {}

  register = async (req: Request, res: Response) => {
    const { email, password } = req.body;
    const user = await this.service.register(email, password);
    res.status(201).json({ id: user.id, email: user.email });
  };

  login = async (req: Request, res: Response) => {
    const { email, password } = req.body;
    const { accessToken, refreshToken } = await this.service.login(email, password);
    res.cookie("refreshToken", refreshToken, COOKIE_OPTS);
    res.json({ accessToken });
  };

  refresh = async (req: Request, res: Response) => {
    const token = req.cookies.refreshToken;
    if (!token) return res.status(401).json({ message: "No refresh token" });
    const accessToken = await this.service.refresh(token);
    res.json({ accessToken });
  };

  logout = async (req: Request, res: Response) => {
    const token = req.cookies.refreshToken;
    if (token) await this.service.logout(token);
    res.clearCookie("refreshToken", { path: "/api/auth/refresh" });
    res.status(204).end();
  };
}

Routes — Layer 1 (wiring only)

modules/auth/auth.routes.ts
import { Router } from "express";
import { PrismaClient } from "@prisma/client";
import { AuthRepository } from "./auth.repository";
import { AuthService } from "./auth.service";
import { AuthController } from "./auth.controller";

const prisma = new PrismaClient();
const controller = new AuthController(new AuthService(new AuthRepository(prisma)));

const router = Router();
router.post("/register", controller.register);
router.post("/login", controller.login);
router.post("/refresh", controller.refresh);
router.post("/logout", controller.logout);

export default router;
Non-negotiable for production

Access token frontend pe sirf ek JS variable / in-memory state mein rakho — kabhi localStorage ya sessionStorage mein nahi. Refresh token kabhi JS ko dikhna hi nahi chahiye, isiliye woh HttpOnly cookie mein hai.

Middleware to protect routes

middlewares/authenticate.ts
import jwt from "jsonwebtoken";
import { Request, Response, NextFunction } from "express";

export function authenticate(req: Request, res: Response, next: NextFunction) {
  const header = req.headers.authorization; // "Bearer "
  const token = header?.split(" ")[1];
  if (!token) return res.status(401).json({ message: "No token" });

  try {
    const payload = jwt.verify(token, process.env.JWT_ACCESS_SECRET!);
    (req as any).user = payload;
    next();
  } catch {
    res.status(401).json({ message: "Token expired or invalid" }); // frontend triggers /refresh
  }
}

Phase 5 · Do this later

Microservices & Performance: Redis + RS256

Kab move karo: app grow ho, microservices add ho, ya DB /refresh hits se slow padne lage.

  • Refresh token ab simple DB UUID nahi — RS256 se signed JWT ban jaata hai (asymmetric keys, private key sirf auth-service ke paas, public key baaki services verify karte hain).
  • Active sessions Postgres/Mongo se Redis mein shift ho jaati hain — lookup <1ms ho jaata hai, main DB app data pe focus karta hai.
modules/auth/auth.repository.ts (Redis version — same interface)
import { Redis } from "ioredis";

export class SessionCacheRepository {
  constructor(private redis: Redis) {}

  async createSession(userId: string, sessionId: string, ttlSeconds: number) {
    await this.redis.set(`session:${sessionId}`, userId, "EX", ttlSeconds);
  }

  async findSession(sessionId: string) {
    return this.redis.get(`session:${sessionId}`);
  }

  async deleteSession(sessionId: string) {
    await this.redis.del(`session:${sessionId}`);
  }
}
// Notice: same method names as AuthRepository -> AuthService doesn't need to change.

Phase 6 · Do this last

Rotation & Automatic Breach Detection

Kab move karo: hazaaron active users, aur advanced token-theft se protect karna ho.

Refresh Token Rotation — har /refresh call pe purana refresh token delete, naya issue. Ek refresh token sirf ek baar use ho sakta hai.

Breach Detection — agar koi purana (already-used/deleted) refresh token dobara use ho, iska matlab woh chura liya gaya tha. Backend turant us user ke saare sessions revoke kar deta hai.

modules/auth/auth.service.ts (rotation)
async refreshWithRotation(oldTokenId: string) {
  const session = await this.repo.findSessionById(oldTokenId);

  if (!session) {
    // token not found = already used/rotated before -> possible theft
    await this.repo.revokeAllSessionsForUnknownToken(oldTokenId);
    throw new Error("REUSE_DETECTED_ALL_SESSIONS_REVOKED");
  }

  await this.repo.deleteSession(oldTokenId);                 // burn the old one
  const newSession = await this.repo.createSession(session.userId, newExpiry());
  const accessToken = signAccessToken(session.userId);

  return { accessToken, refreshToken: newSession.id };
}

Before you deploy Phase 4

Ship Checklist

  • Access token 15 min expiry, memory mein stored (kabhi localStorage nahi)
  • Refresh token 7 din expiry, HttpOnly + Secure + SameSite=Strict cookie mein
  • Session table DB mein — valid refresh tokens track karne ke liye
  • Passwords bcrypt (cost ≥ 12) se hashed, kabhi plaintext nahi
  • Logout = DB se session delete + cookie clear, dono
  • Har protected route authenticate middleware se guard hai
  • 5-layer separation maintained — controller mein koi business logic nahi
  • HTTPS enforced production mein (varna Secure cookie kaam hi nahi karega)
Phase 1 → 6, ek hi 5-layer structure ke andar. Sirf implementation deep hoti jaati hai — architecture nahi todhna padta.
